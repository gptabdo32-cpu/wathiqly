# المواصفات التقنية لميزة المساعد القانوني الذكي للعقود (AI Escrow Arbitrator)

## 1. المقدمة

تهدف هذه الوثيقة إلى تحديد المواصفات التقنية لميزة "المساعد القانوني الذكي للعقود" (AI Escrow Arbitrator) ضمن منصة "وثّقلي". ستعمل هذه الميزة على دمج نموذج لغوي متخصص (Legal LLM) لتحليل شروط العقود الذكية (Escrow) وتقديم تقييمات حول عدالتها وقانونيتها، مما يقلل من النزاعات المحتملة ويزيد من الشفافية والثقة بين الأطراف.

## 2. الأهداف

*   **تحليل العقود**: تحليل شروط العقد الذكي تلقائيًا قبل تفعيله.
*   **تحديد الثغرات القانونية**: الكشف عن أي ثغرات أو بنود غير عادلة أو غير قانونية في شروط العقد.
*   **تقديم توصيات**: توفير توصيات للطرفين لتحسين صياغة العقد وضمان عدالته.
*   **تقليل النزاعات**: المساهمة في تقليل النزاعات بنسبة تصل إلى 90% من خلال التنبيه المبكر للمشكلات.
*   **الامتثال للمعايير**: ضمان توافق الميزة مع المعايير الهندسية والأمنية والتقنية العالمية، واستخدام Manus Forge API.

## 3. معمارية النظام

ستتكون الميزة من المكونات الرئيسية التالية:

1.  **واجهة المستخدم (Frontend)**: لعرض نتائج التحليل القانوني والتوصيات للمستخدمين.
2.  **الخدمة الخلفية (Backend Service)**: ستكون مسؤولة عن استدعاء نموذج اللغة، معالجة البيانات، وتخزين نتائج التحليل.
3.  **نموذج اللغة المتخصص (Legal LLM)**: سيتم استدعاؤه عبر Manus Forge API لتحليل النصوص القانونية.
4.  **قاعدة البيانات (Database)**: لتخزين نتائج تحليل العقود بواسطة AI Arbitrator.

### 3.1. تدفق البيانات (Data Flow)

سيكون تدفق البيانات كالتالي:

1.  يقوم المستخدم بإنشاء أو تعديل عقد ذكي (Escrow) عبر واجهة المستخدم.
2.  يتم إرسال تفاصيل العقد إلى الخدمة الخلفية.
3.  تقوم الخدمة الخلفية باستدعاء وظيفة جديدة `analyzeEscrowContract`.
4.  تقوم `analyzeEscrowContract` بتجهيز نص العقد وإرساله إلى Legal LLM عبر `invokeLLM` (الموجود في `server/_core/llm.ts`).
5.  يقوم Legal LLM بتحليل العقد وإرجاع النتائج (نقاط الضعف، التوصيات، مستوى المخاطر القانونية، درجة العدالة).
6.  تقوم `analyzeEscrowContract` بتخزين هذه النتائج في جدول `ai_arbitrator_analysis` الجديد.
7.  يتم عرض نتائج التحليل للمستخدم في واجهة المستخدم.

## 4. تصميم قاعدة البيانات

تم إنشاء جدول `ai_arbitrator_analysis` في `drizzle/schema_ai_arbitrator.ts` لتخزين نتائج تحليل العقود. تفاصيل الجدول:

| الحقل              | النوع                                   | الوصف                                                                |
| :----------------- | :--------------------------------------- | :------------------------------------------------------------------- |
| `id`               | `int` (مفتاح أساسي، ترقيم تلقائي)        | معرف فريد لكل تحليل.                                                |
| `escrowId`         | `int` (غير فارغ)                         | معرف العقد الذكي المرتبط بهذا التحليل.                               |
| `analysisVersion`  | `varchar(50)` (افتراضي: "1.0.0")        | إصدار نموذج التحليل المستخدم.                                        |
| `fairnessScore`    | `int` (غير فارغ)                         | درجة العدالة للعقد (0-100).                                          |
| `legalRiskLevel`   | `enum` (`low`, `medium`, `high`, `critical`) (غير فارغ) | مستوى المخاطر القانونية للعقد.                                       |
| `analysisResults`  | `json` (غير فارغ)                        | نتائج التحليل التفصيلية (الثغرات، التوصيات، تحليل البنود).           |
| `summary`          | `text` (غير فارغ)                        | ملخص للتحليل باللغة العربية.                                        |
| `status`           | `enum` (`pending`, `completed`, `failed`) (افتراضي: `pending`) | حالة عملية التحليل.                                                 |
| `modelUsed`        | `varchar(100)`                           | اسم نموذج اللغة المستخدم للتحليل.                                    |
| `tokensUsed`       | `int`                                    | عدد الرموز (tokens) المستخدمة في عملية التحليل.                      |
| `createdAt`        | `timestamp` (افتراضي: `now()`)          | تاريخ ووقت إنشاء سجل التحليل.                                        |
| `updatedAt`        | `timestamp` (افتراضي: `now()`، تحديث تلقائي) | تاريخ ووقت آخر تحديث لسجل التحليل.                                   |

## 5. تكامل نموذج اللغة (LLM Integration)

سيتم استخدام `server/_core/llm.ts` لاستدعاء Manus Forge API. يجب أن يتم تمرير نص العقد إلى النموذج مع تعليمات واضحة لتحليل الجوانب القانونية والعدالة. سيتم استخدام `gemini-2.5-flash` كنموذج افتراضي.

### 5.1. هيكل الاستدعاء (Prompt Structure)

سيتم بناء الـ prompt لنموذج اللغة لطلب تحليل شامل للعقد. سيتضمن الـ prompt ما يلي:

*   **التعليمات**: توجيه النموذج لتحليل العقد من منظور قانوني، مع التركيز على العدالة، الثغرات المحتملة، والبنود غير الواضحة.
*   **نص العقد**: النص الكامل للعقد المراد تحليله.
*   **تنسيق الإخراج**: طلب إخراج بتنسيق JSON محدد يتضمن `fairnessScore`, `legalRiskLevel`, `loopholes`, `recommendations`, `clauses_analysis`, و `summary`.

مثال على هيكل الإخراج المتوقع من LLM:

```json
{
  "fairnessScore": 85,
  "legalRiskLevel": "low",
  "loopholes": [
    "عدم وجود بند واضح لآلية فض النزاعات في حالة عدم التوافق على تسليم الخدمة."
  ],
  "recommendations": [
    "إضافة بند يحدد بوضوح آلية فض النزاعات، مثل التحكيم أو الوساطة.",
    "تحديد معايير واضحة لقبول أو رفض الخدمة/المنتج."
  ],
  "clauses_analysis": [
    {
      "clause": "يوافق الطرفان على أن الدفع سيتم عند اكتمال الخدمة.",
      "status": "fair",
      "comment": "بند واضح ومقبول."
    },
    {
      "clause": "يحق للبائع إنهاء العقد في أي وقت دون إشعار مسبق.",
      "status": "unfair",
      "comment": "هذا البند يضع المشتري في موقف ضعيف وقد يؤدي إلى نزاعات."
    }
  ],
  "summary": "العقد بشكل عام عادل، ولكن ينقصه بعض البنود المتعلقة بفض النزاعات وحقوق الإنهاء."
}
```

## 6. واجهات برمجة التطبيقات (API Endpoints)

سيتم إضافة واجهات API جديدة ضمن `smartEscrowRouter` في `server/routers/smartEscrow.ts`.

### 6.1. `analyzeEscrowContract` (Protected Procedure)

*   **الوصف**: تستدعي هذه الواجهة نموذج اللغة لتحليل عقد ذكي معين وتخزن النتائج.
*   **المدخلات**: `escrowId: number` (معرف العقد).
*   **المخرجات**: `success: boolean`, `analysisId: number`.
*   **المنطق**: 
    1.  التحقق من صلاحيات المستخدم (يجب أن يكون المشتري أو البائع للعقد).
    2.  جلب تفاصيل العقد من قاعدة البيانات.
    3.  بناء الـ prompt لنموذج اللغة باستخدام تفاصيل العقد.
    4.  استدعاء `invokeLLM` لتحليل العقد.
    5.  تخزين نتائج التحليل في جدول `ai_arbitrator_analysis`.
    6.  تسجيل العملية في `auditLogs`.

### 6.2. `getEscrowAnalysis` (Protected Procedure)

*   **الوصف**: تسترجع نتائج التحليل القانوني لعقد ذكي معين.
*   **المدخلات**: `escrowId: number`.
*   **المخرجات**: `AiArbitratorAnalysis` object أو `null`.
*   **المنطق**: 
    1.  التحقق من صلاحيات المستخدم.
    2.  جلب أحدث تحليل من جدول `ai_arbitrator_analysis` لـ `escrowId` المحدد.

## 7. الاعتبارات الأمنية والهندسية

*   **أمان البيانات**: يجب تشفير أي بيانات حساسة يتم تمريرها إلى LLM أو تخزينها في قاعدة البيانات (مثل `description` إذا احتوت على معلومات شخصية). حاليًا، `description` في جدول `escrows` هو `text`، لذا يجب التأكد من عدم تخزين معلومات حساسة فيه مباشرة أو تطبيق التشفير عند الحاجة.
*   **التحقق من الصلاحيات**: سيتم تطبيق التحقق الصارم من صلاحيات المستخدم قبل السماح بتحليل العقد أو عرض نتائجه.
*   **التعامل مع الأخطاء**: يجب أن تكون استدعاءات LLM محاطة بآليات قوية للتعامل مع الأخطاء (مثل مهلة الاستجابة، أخطاء API).
*   **قابلية التوسع**: تصميم الميزة لتدعم التوسع المستقبلي، مع الأخذ في الاعتبار أن استدعاءات LLM قد تكون مكلفة من حيث الموارد والوقت.
*   **التسجيل والمراقبة**: تسجيل جميع استدعاءات LLM ونتائج التحليل في `auditLogs` للمراجعة والمراقبة.
*   **التحقق من صحة المدخلات**: استخدام `zod` للتحقق من صحة المدخلات لجميع واجهات API الجديدة.

## 8. الخطوات التالية

1.  تحديث `drizzle/schema.ts` لاستيراد `schema_ai_arbitrator.ts`.
2.  إنشاء ملف `server/db_ai_arbitrator.ts` لتضمين وظائف قاعدة البيانات الخاصة بـ `aiArbitratorAnalysis`.
3.  تعديل `server/routers/smartEscrow.ts` لإضافة واجهات API الجديدة (`analyzeEscrowContract`, `getEscrowAnalysis`).
4.  تعديل واجهة المستخدم (Frontend) لاستدعاء واجهات API الجديدة وعرض النتائج.
