# تقرير تنفيذ طبقة التنفيذ (Execution Layer) لمنصة Wathiqly

تم بنجاح تحويل النظام من نظام "آمن البيانات ولكن غير مكتمل التنفيذ" إلى نظام "موثوق بالكامل، ذاتي الإصلاح، وجاهز للإنتاج".

## 1. المكونات التي تم تنفيذها

| المكون | الوصف | الفوائد التقنية |
| :--- | :--- | :--- |
| **Outbox Worker** | محرك معالجة الأحداث المخزنة في قاعدة البيانات. | يضمن تنفيذ الأحداث بترتيبها الصحيح وبدون فقدان، حتى في حالة تعطل الخادم. |
| **Retry System** | نظام إعادة المحاولة باستخدام **Exponential Backoff**. | يمنع إغراق النظام بالطلبات الفاشلة ويوفر وقت استراحة متزايد للأنظمة الخارجية للتعافي. |
| **Dead Letter Queue (DLQ)** | نظام إدارة الأحداث التي فشلت نهائياً. | يسمح للمسؤولين بمراجعة الأحداث الفاشلة وإعادة تشغيلها يدوياً بعد إصلاح المشكلة. |
| **Event Handler Execution Model** | نموذج تنفيذ المعالجات (Subscribers). | فصل عملية النشر عن التنفيذ، مما يسمح بمعالجة غير متزامنة وقابلة للتوسع. |
| **Saga Recovery Engine** | محرك استرداد الـ Sagas العالقة. | يكتشف العمليات الطويلة (مثل الضمان) التي توقفت بسبب خلل فني ويستأنفها من آخر نقطة نجاح. |

## 2. التفاصيل التقنية الدقيقة

### استراتيجية القفل (Locking Strategy)
استخدمنا استراتيجية `FOR UPDATE SKIP LOCKED` في SQL. تضمن هذه الاستراتيجية:
- **عدم حدوث Race Conditions**: لا يمكن لعاملين معالجة نفس الحدث في نفس الوقت.
- **قابلية التوسع (Horizontal Scaling)**: يمكن تشغيل عدة نسخ من `OutboxWorker` لزيادة سرعة المعالجة دون تداخل.

### صيغة إعادة المحاولة (Retry Formula)
تم تطبيق صيغة التأخير الأسي:
`Delay = BaseDelay (2s) * 2^retries`
هذا يعني أن التأخير يتضاعف مع كل محاولة فاشلة (2ث، 4ث، 8ث، 16ث...) حتى الوصول إلى الحد الأقصى (10 محاولات).

### استمرارية الـ Saga
محرك الاسترداد يقوم بمسح جدول `saga_states` كل 30 ثانية للبحث عن أي Saga في حالة `STARTED` أو `PROCESSING` ولم يتم تحديثها منذ أكثر من 5 دقائق، مما يضمن عدم ضياع أي معاملة مالية.

## 3. سيناريوهات الفشل وكيفية التعامل معها

1. **نجاح الـ DB Commit وتعطل الـ Worker**: الحدث يظل بحالة `pending` في قاعدة البيانات. عند إعادة تشغيل الـ Worker، سيجده ويعالجه.
2. **تعطل الـ Worker أثناء التنفيذ**: الحدث يظل بحالة `processing`. محرك الاسترداد أو الـ Worker (بعد فترة زمنية) سيعيد ضبطه أو معالجته بناءً على `lastAttemptAt`.
3. **فشل معالج الحدث جزئياً**: يتم تسجيل الخطأ، وزيادة عدد المحاولات، وإعادة المحاولة لاحقاً. بفضل `IdempotencyManager` لن تتكرر الآثار الجانبية الناجحة.
4. **فشل خطوة الـ Saga بعد الكتابة في DB**: الـ Saga Recovery Engine سيكتشف التوقف ويستأنف الخطوة.
5. **إعادة تشغيل النظام بعد التوقف**: تبدأ جميع المحركات تلقائياً بمسح قاعدة البيانات ومعالجة كل ما فات.

## 4. تدفق التنفيذ (Execution Flow)

**قبل التعديل (Write-only):**
`UseCase -> Transaction -> Write Data -> Write Outbox -> Commit -> (End)`

**بعد التعديل (Reliable Execution):**
1. `UseCase -> Transaction -> Write Data -> Write Outbox -> Commit`
2. `OutboxWorker -> Poll (Locked) -> Push to Queue`
3. `EventQueue (BullMQ) -> Worker -> ExecuteHandlers`
4. `Handler -> Business Logic -> (Success/Fail)`
5. `Worker -> Update Outbox Status (Completed/Failed)`
6. `SagaRecoveryEngine -> (Optional) Resume if stuck`
