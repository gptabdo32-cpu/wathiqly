# تحليل المعمارية وحدود الاتساق (Architecture & Consistency Boundaries)

بناءً على مبادئ النظام المالي الحتمي (Deterministic Financial System)، تم تحديث القواعد التالية لضمان سلامة البيانات وفصل المسؤوليات.

## 1. حدود الاتساق (Consistency Boundaries)

- **ACID (داخل الخدمة)**: يتم ضمان الاتساق القوي باستخدام المعاملات (Transactions) داخل حدود الخدمة الواحدة (مثل `EscrowService` أو `PaymentService`).
- **الاتساق النهائي (عبر الخدمات)**: يتم التنسيق بين الخدمات المختلفة (مثل الضمان والدفع) باستخدام **الأحداث (Events)** ونمط **Saga**.

| النطاق | نوع الاتساق | الآلية المستخدمة |
| :--- | :--- | :--- |
| تحديث حالة الضمان محلياً | ACID | `TransactionManager.run` |
| حجز الأموال في نظام الدفع | ACID | `PaymentService.lockFunds` |
| التنسيق بين الضمان والدفع | Eventual | `EscrowSaga` + `Outbox Pattern` |

## 2. إزالة الغموض المعماري (Removing Ambiguity)

تم اتخاذ القرارات التالية لتوحيد منطق العمل:

1. **توحيد التنسيق (Orchestration)**:
   - سيتم استخدام **Saga** حصرياً للعمليات التي تتطلب التنسيق بين خدمات متعددة (مثل إنشاء الضمان الذي يتطلب حجز أموال).
   - سيتم استخدام **Use Cases** للعمليات البسيطة داخل حدود الخدمة الواحدة.
   - **القاعدة**: لا يجوز استخدام كليهما لنفس التدفق.

2. **ملكية المنطق (Logic Ownership)**:
   - **Domain**: يحتوي على قواعد العمل الصرفة (مثل: هل يمكن تحرير الضمان؟).
   - **Application**: يحتوي على تنسيق العمليات (مثل: استدعاء المستودع، استدعاء خدمة الدفع).
   - **Infrastructure**: يحتوي على تفاصيل التنفيذ (مثل: Drizzle ORM، استدعاء APIs خارجية).

## 3. معالجة الانتهاكات السابقة (Addressing Violations)

| الانتهاك | الإجراء التصحيحي |
| :--- | :--- |
| تسريب منطق البنية التحتية | نقل جميع عمليات `db.update` إلى المستودعات (Repositories). |
| الاعتماد المباشر على التنفيذ | استخدام واجهات (Interfaces) لجميع الخدمات الخارجية والعمال (Workers). |
| منطق العمل خارج الدومين | نقل قواعد التحقق من الحالة إلى `Escrow` و `Order` entities. |
| الدقة المالية | استبدال `parseFloat` بـ `Decimal.js` لضمان عدم فقدان الدقة في العمليات الحسابية. |

## 4. خطة العمل القادمة

- دمج منطق `CreateEscrow` و `EscrowSaga` في مكون واحد منسق.
- التأكد من أن كل عملية مالية لها `CorrelationId` فريد لتتبعها عبر الخدمات.
- فرض استخدام `Outbox Pattern` لجميع الأحداث الصادرة لضمان عدم فقدانها.
