# تقرير التحليل الأمني والهندسي لميزة "وثّقلي ID & Pay Wallet"

## 1. مقدمة
يهدف هذا التقرير إلى تقديم تحليل شامل للكود الحالي لميزة المحفظة المتكاملة، مع التركيز على نقاط القوة والضعف ومقارنتها بالمعايير العالمية مثل **OWASP Top 10** و **PCI-DSS**. بناءً على هذا التحليل، سيتم اقتراح تحسينات أمنية وهندسية لرفع مستوى الحماية والموثوقية.

## 2. تحليل الوضع الحالي

### نقاط القوة الهندسية

| الميزة | الوصف | الفائدة الهندسية |
| :--- | :--- | :--- |
| **المعاملات الذرية (Atomic)** | استخدام `db.transaction` في جميع العمليات المالية (تحويل P2P، دفع الفواتير). | **ضمان سلامة البيانات (Data Integrity)**: تمنع هذه الآلية حدوث أخطاء في الأرصدة، حيث يتم تنفيذ العملية بالكامل أو إلغاؤها بالكامل. |
| **قفل قاعدة البيانات (DB Locking)** | استخدام `.for("update")` عند تحديث أرصدة المحافظ. | **منع حالات التسابق (Race Conditions)**: تضمن عدم إجراء عمليتين على نفس المحفظة في نفس الوقت، مما يحمي من أخطاء الرصيد. |
| **سجلات التدقيق (Audit Logs)** | تسجيل جميع العمليات المالية الهامة في جدول `walletAuditLogs`. | **التتبع والشفافية (Traceability & Transparency)**: يوفر سجلاً كاملاً غير قابل للتعديل لكل حركة مالية، وهو متطلب أساسي لمعيار **ISO 27001**. |
| **فصل الاهتمامات (Separation of Concerns)** | بناء الميزة في ملفات منفصلة (`schema_wallet_id.ts`, `wallet_id.ts`) ودمجها في النظام. | **الصيانة والتطوير (Maintainability)**: يسهل هذا التنظيم إضافة ميزات جديدة أو تعديل الميزات الحالية دون التأثير على باقي أجزاء النظام. |

### نقاط الضعف الأمنية المكتشفة

| الثغرة الأمنية (OWASP) | الوصف | التأثير المحتمل |
| :--- | :--- | :--- |
| **A01: Broken Access Control** | لا يوجد تحقق كافٍ من ملكية المحفظة عند إجراء عمليات السحب أو الإيداع. | يمكن لمستخدم مخترق أن يستدعي نقاط النهاية (Endpoints) مباشرة بمعرف محفظة مستخدم آخر. |
| **A05: Security Misconfiguration** | عدم وجود آلية للحد من عدد الطلبات (Rate Limiting) على نقاط النهاية الحساسة. | **هجمات التخمين (Brute-force attacks)**: يمكن للمهاجم تخمين رموز التحقق الثنائي (2FA) أو استغلال نقاط النهاية لإرسال رسائل مزعجة. |
| **A02: Cryptographic Failures** | عدم وجود آلية لمنع إعادة تنفيذ الطلبات (Replay Attacks) في العمليات المالية. | **تكرار المعاملات (Duplicate Transactions)**: يمكن للمهاجم اعتراض طلب تحويل مالي وإعادة إرساله مرة أخرى لتكرار العملية بشكل غير شرعي. |
| **A03: Injection** | الاعتماد على `Math.random()` لإنشاء معرفات فريدة للمعاملات. | **تصادم المعرفات (ID Collisions)**: قد يؤدي إلى تداخل البيانات وصعوبة تتبع المعاملات بشكل دقيق، وهو ما يتعارض مع متطلبات **PCI-DSS**. |
| **A01: Broken Access Control** | نظام الهوية الرقمية (SSO) لا يطبق معيار **PKCE** في بروتوكول OAuth 2.0. | **اعتراض رمز التفويض (Authorization Code Interception)**: في التطبيقات العامة (مثل تطبيقات الهاتف)، يمكن للمهاجم سرقة رمز التفويض واستخدامه للوصول لبيانات المستخدم. |

## 3. خطة التحسينات الأمنية
بناءً على التحليل أعلاه، تم وضع خطة عمل لتطبيق التحسينات التالية:

1.  **تطبيق Rate Limiting**: إضافة طبقة حماية للحد من عدد الطلبات على نقاط النهاية الحساسة مثل `sendMoney` و `payBill`.
2.  **آلية Idempotency Key**: منع إعادة تنفيذ الطلبات المالية عبر إضافة مفتاح فريد لكل طلب (Idempotency-Key) يتم التحقق منه في الخادم.
3.  **تعزيز OAuth 2.0**: تطبيق آلية **PKCE (Proof Key for Code Exchange)** لزيادة أمان تدفق التفويض في نظام الهوية الرقمية الموحدة.
4.  **استخدام UUIDs**: استبدال `Math.random()` بمعرفات فريدة عالمياً (UUIDs) لضمان عدم تكرار معرفات المعاملات.

سيتم البدء في تنفيذ هذه التحسينات في المراحل التالية لضمان أعلى مستويات الأمان والموثوقية للمنصة.
